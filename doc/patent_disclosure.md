## 四、背景技术

### 4.1 现有技术及其不足

在 Web 应用的测试领域，测试用例的编写是一项高度依赖人工经验的工作。当前存在以下几类自动化辅助手段：

#### （1）操作录制 → 脚本生成类工具

例如 Selenium IDE、Playwright Codegen、Cypress Studio 等。此类工具通过录制用户的浏览器操作，直接生成可执行的自动化测试脚本（代码）。

**不足之处：**

- **生成结果是代码而非测试用例**：输出的是程序员可读的自动化脚本，而非测试工程师需要的自然语言测试步骤与预期结果描述。从测试管理角度，团队仍需人工编写业务层面的测试用例文档。
- **录制与理解耦合**：脚本生成在录制阶段就完成了对操作的"理解"（选择定位器、推断断言），一旦推断错误，需要重新录制。录制产物不可复用、不可迭代优化。
- **缺乏 UI 状态证据**：这些工具仅记录操作序列本身，不采集操作前后的页面结构化状态，因此无法回答"操作引起了什么变化"这一核心测试问题。

#### （2）AI 辅助测试平台

例如 Testim、Mabl、Applitools 等。此类平台利用机器学习来辅助元素定位的自愈（self-healing）、视觉对比测试等。

**不足之处：**

- **仍然是脚本级别的自动化**：AI 的能力被用于提升脚本的稳定性（如元素定位自愈），而非用于生成业务层面的测试用例文档。
- **依赖视觉截图对比**：视觉对比方法对 UI 变化的描述停留在像素级差异，无法提供语义级别的"页面结构变化"信息。
- **缺乏结构化证据链**：AI 做出判断的依据不透明，用户难以验证和调试。

#### （3）RPA（机器人流程自动化）录制

例如 UiPath、Microsoft Power Automate 等。此类工具录制桌面或浏览器操作用于流程自动化回放。

**不足之处：**

- **目的是回放而非测试**：RPA 的目标是重复执行操作流程，而非生成带有验证点（预期结果/UI 变化）的测试用例。
- **不采集状态快照**：缺乏操作前后的结构化页面状态对比能力。

#### （4）现有技术的共性缺陷

综合上述现有技术，存在以下共性缺陷：

1. **录制与翻译不解耦**：录制阶段即完成了对操作的理解和输出生成，不允许对同一份录制数据做迭代优化。
2. **缺乏操作前后的结构化 UI 状态采集**：没有系统性地采集操作前后的页面可访问性树（Accessibility Tree）快照，无法为后续分析提供精确的结构化证据。
3. **未利用大语言模型的归纳推理能力**：现有工具未将 LLM 用于从结构化操作证据中归纳自然语言测试用例。
4. **快照采集存在竞态问题**：在浏览器事件处理器内异步调用快照接口会导致"操作前快照被同步变化污染"或"操作后快照未包含渐进行为"的竞态问题，现有技术缺乏系统性的解决方案。

### 4.2 本发明需要解决的技术问题

1. 如何在浏览器操作录制过程中，系统性地采集每个操作前后的精确 UI 结构化状态快照，同时避免异步快照调用引发的竞态问题？
2. 如何在不让大语言模型自行对比大段文本的前提下，通过预处理构建结构化证据包，让 AI 基于可靠证据生成准确的自然语言测试用例？
3. 如何实现录制阶段与翻译阶段的完全解耦，使同一份录制数据可以反复调参翻译，无需重新录制？

## 六、技术方案

### 6.1 系统总体架构

本发明的系统由三个核心子系统组成，通过文件系统解耦协作：

```
┌─────────────────────────────────────────────────────────────┐
│                     用户在浏览器中操作                         │
└─────────────┬───────────────────────────────────────────────┘
              ▼
┌─────────────────────────────────────────────────────────────┐
│              录制子系统（Recorder）                            │
│  ┌────────────────┐  ┌──────────────┐  ┌─────────────────┐  │
│  │ 浏览器注入脚本  │  │ 快照轮询引擎  │  │  分文件存储引擎  │  │
│  │ (事件捕获 +    │  │ (周期AX快照   │  │  (actions/ +    │  │
│  │  formState     │  │  + 缓存)      │  │  snapshots/ +   │  │
│  │  同步采集)     │  │              │  │  meta.json)     │  │
│  └────────────────┘  └──────────────┘  └─────────────────┘  │
└─────────────┬───────────────────────────────────────────────┘
              ▼  文件系统（output/run_*/）
┌─────────────────────────────────────────────────────────────┐
│              翻译子系统（Translator）                          │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │        预处理器（Preprocessor）                           │ │
│  │  语义归并 → 快照差异 → 上下文提取 → 表单增量 → 分类标注    │ │
│  └─────────────┬───────────────────────────────────────────┘ │
│                ▼                                             │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │        AI 工作流（Workflow）                               │ │
│  │  Phase 1: 逐条分析（可恢复） → Phase 2: 归纳用例           │ │
│  └─────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
              ▼
        AI_steps.md（逐条步骤分析）
        AI_cases.md（结构化测试用例）
```

**关键设计原则：**

- **证据优先（Evidence > Guess）**：系统在架构上强制"先把证据算出来，再让 AI 基于证据写结论"，AI 被约束为不得在信息不足时猜测。
- **稳定采集 / 可变理解**：录制阶段只做稳定的原始数据采集；翻译阶段允许频繁调整预处理策略和 Prompt 模板，两阶段完全解耦。
- **文件系统解耦**：录制产物与翻译产物通过标准化的目录结构交互，同一份录制数据可反复翻译。

### 6.1.1 术语与数据对象定义（便于理解与撰写权利要求）

为避免“实现细节”与“权利要求范围”混淆，本文将关键术语定义如下（并非对实现语言/框架的限定）：

1. **操作记录（action）**：对用户一次物理操作的结构化记录，至少包含操作类型、目标元素的轻量特征、页面标识信息、时间戳等。
2. **页面结构快照（snapshot）**：对操作发生前或发生后的页面结构化状态的文本化表示，优选来源于页面无障碍树（Accessibility Tree）或与其等价的语义结构树。
3. **表单状态快照（formStateDelta）**：在操作发生瞬间同步采集到的表单控件状态集合，至少包含表单控件的值/勾选状态/选择状态等。
4. **证据包（enrichedAction）**：对单条操作记录的富化结果，至少包含快照差异、表单增量、上下文片段、分类提示中的一项或多项，用于作为大语言模型推理的输入证据。
5. **间隙归属原则**：将相邻两次操作之间发生的页面变化归属于前一次操作，用于定义操作后快照的边界。

### 6.2 录制子系统的技术方案

#### 6.2.1 物理操作捕获机制

本发明在录制阶段仅捕获用户的物理操作（physical user actions），而非所有 DOM 事件。通过浏览器上下文（Browser Context）级别注入 JavaScript 脚本，在目标页面的 `document` 上以 `capture` 阶段注册事件监听器：

- **鼠标操作**：`click`（单击）、`dblclick`（双击）、`contextmenu`（右键）
- **键盘操作**：`keydown` 事件中的 `Enter`、`Tab`、`Escape`、`Space` 键

**不捕获的事件类型**（与现有技术的重要区别）：

- 不监听 `change`、`input`、`hover`、`drag`、`MutationObserver` 等事件
- 文本输入的具体内容不通过逐字符监听获取，而是通过本发明独创的**双通道证据机制**（快照差异 + 表单状态增量）在翻译阶段推导得出

每个被捕获的操作记录包含以下结构化信息：
- 操作类型（type）
- 目标元素的轻量描述（tag、id、name、type、text、label、placeholder、selector）
- 操作坐标（position）
- 页面 URL 和标题
- 时间戳（timestamp）
- 表单状态快照（formStateDelta，详见 6.2.3）

操作数据通过浏览器自动化控制框架提供的“页面端调用宿主端回调函数”的机制，从浏览器端回传至宿主端进程。注入脚本优选实现跨 iframe 的回调函数查找机制（当前 window → parent → top），以确保嵌套框架中的操作也能被捕获。

#### 6.2.2 周期轮询快照与混合策略保存（核心创新点）

**要解决的技术问题：**

对每个用户操作，系统需要一张"操作前快照"（preSnapshot，反映操作前的页面状态）和一张"操作后快照"（postSnapshot，反映操作后包含渐进行为的页面状态）。然而，浏览器无障碍树快照接口（`page.accessibility.snapshot()`）是异步调用，在事件处理器内直接调用会导致以下竞态问题：

1. **操作前快照污染**：在 click 事件处理器内 `await snapshot()` 时，同步 DOM 变化（如 toggle 翻转）可能已经发生，导致拍到的"操作前快照"实际已包含操作效果。
2. **操作后快照不完整**：操作触发的渐进行为（用户继续打字、异步数据加载、动画结算）尚未完成时就拍摄了"操作后快照"。

**本发明的解决方案 —— 周期轮询快照与混合策略保存：**

本发明将快照的"拍摄"与"保存"解耦为两个独立过程：

**（一）周期轮询拍摄（解决操作前快照污染问题）**

在宿主端进程中设置周期性定时器（例如每 300 毫秒一次），持续调用页面结构快照接口（优选为无障碍树快照接口），将获取到的最新快照缓存于内存中的“快照缓存单元”。当用户操作到达时，事件处理器**不在处理器内执行异步快照调用**，而是直接读取所述快照缓存单元的缓存快照作为证据输入。由于缓存快照在上一次轮询周期中生成，早于当前操作触发，因此不会被当前操作的同步页面变化所污染。

**（二）混合策略保存（解决操作后快照不完整问题 + 零冗余存储）**

本发明采用"间隙归属原则"：两个相邻用户操作之间发生的一切变化（用户继续打字、异步渲染、动画结算等），归属于前一个操作。由此推导出核心命名约定：

> **`post(N) === pre(N+1)`**：第 N 个操作的操作后快照，与第 N+1 个操作的操作前快照是同一张快照。

据此，N 个操作只需存储 N+1 张快照（零冗余）。保存策略如下：

- **录制开始时**：拍摄并保存 `snapshot_000` 作为初始态（第 1 个操作的 preSnapshot）
- **第 1 个操作到达时**：仅保存操作数据（`action_001.json`），不保存快照（其 preSnapshot = `snapshot_000`，已保存）
- **第 2 个及以后的操作到达时**：将当前内存缓存的快照保存为 `snapshot_{N-1}`。由于该缓存快照经过了若干个轮询周期（涵盖前一操作到当前操作之间的时间间隙），已包含前一操作触发的渐进行为（打字、异步渲染等），因此它同时是前一操作的完整 postSnapshot 和当前操作的干净 preSnapshot。
- **录制结束时**：补保存终态快照 `snapshot_N`（最后一个操作的 postSnapshot）

**该方案的技术效果：**

1. **操作前快照干净**：缓存快照来自定时轮询，不在操作事件处理器内执行异步调用，避免了同步变化的污染。
2. **操作后快照完整**：利用"下一个操作到达"作为"前一操作的渐进行为已基本结算"的代理信号，确保操作后快照包含了充分的 UI 变化。
3. **零冗余存储**：N 个操作仅需 N+1 张快照，相邻操作共享中间快照。
4. **强可追溯性**：任意操作 N 都可精确定位其 preSnapshot（`snapshot_{N-1}`）和 postSnapshot（`snapshot_{N}`），以及两者的差异（`diff_N`）。

**时序图说明：**

```
时间轴 →
                                                    
后台轮询: ──●────────●────────●────────●────────●────────●──
           S₀      S₁       S₂       S₃       S₄      S₅
                                                    
用户操作:       action_1         action_2          action_3
                 ↓                 ↓                 ↓
                                                    
磁盘保存: [S₀]            [S₁=缓存]         [S₂=缓存]    [S₃=终态]
          ↑start()        ↑action_2到达     ↑action_3到达  ↑stop()
          =pre(1)         =post(1)=pre(2)  =post(2)=pre(3)=post(3)
```

其中：
- `●` 表示一次轮询拍摄（结果更新到内存缓存）
- `[Sₙ]` 表示一次磁盘保存
- action_1 到达时不保存快照（pre(1)=S₀ 已在 start 时保存）
- action_2 到达时保存当前缓存（该缓存包含 action_1 后的渐进行为）

#### 6.2.3 表单状态同步采集机制（formStateDelta）

周期轮询快照存在固有的时间粒度限制（采样间隔为百毫秒级），且浏览器无障碍树对部分表单控件值的反映存在不稳定性。为解决这一问题，本发明设计了第二条独立的证据采集通道：

在浏览器端注入脚本中，于 `pointerdown` 和 `keydown` 事件的 **`capture` 阶段**（先于目标阶段和冒泡阶段执行），**同步地**遍历当前页面所有表单元素（`input`、`select`、`textarea`、带有 ARIA 状态属性的元素），读取其精确的当前状态值（`value`、`checked`、`selectedIndex`、`aria-expanded`、`aria-checked`、`aria-selected`），以 CSS 选择器为键构建状态快照对象 `formStateDelta`，随操作数据一起发送。

**同步采集的关键意义：**

- `capture` 阶段在 DOM 的同步变化之前执行（`pointerdown` 早于 `click`），保证读取到的是操作发生前的真实表单状态。
- 这与 6.2.2 中的异步轮询快照形成互补：轮询快照提供全页面结构化状态，formStateDelta 提供表单元素的精确瞬时值。两条证据通道在翻译阶段独立对比，不在采集阶段合并，保持"证据来源分离"原则。

**formStateDelta 在后续翻译中的作用：**

- 通过比较相邻操作的 formStateDelta，可精确计算出"两次操作之间用户输入了什么"，这是实现"输入识别"语义归并的核心数据依据。

#### 6.2.4 无障碍树快照裁剪与文本化

原始的浏览器无障碍树（Accessibility Tree）包含大量无意义节点且层级过深。本发明对原始快照进行以下处理：

1. **深度裁剪**：限制最大递归深度（默认 8 层），超出部分丢弃。
2. **无意义节点过滤**：将角色（role）为 `none`、`generic`、`presentation`、`LineBreak`、`InlineTextBox`、`StaticText` 的叶子节点（无 name、无 value、无子节点）过滤掉。
3. **属性精简**：仅保留有值的属性（role、name、value、checked、pressed、expanded、selected、disabled、required、level）。
4. **YAML 风格文本化**：将裁剪后的树结构转为缩进表示的文本格式（每行格式：`- role "name" [attributes]`，缩进表示层级关系），该格式相较于 JSON 可显著减少输入长度，且大语言模型更容易理解层级结构。

#### 6.2.5 数据契约与存储规范

每次录制生成一个带时间戳的独立目录 `output/run_<timestamp>/`，包含以下标准化产物：

```
output/run_<timestamp>/
  meta.json                 # 录制元信息 + 操作摘要索引
  snapshots/
    snapshot_000.txt        # 初始态
    snapshot_001.txt        # post(action_1) = pre(action_2)
    ...
    snapshot_N.txt          # 终态
  actions/
    action_001.json         # 第 1 个操作（含 formStateDelta）
    action_002.json
    ...
    action_N.json
```

关键命名约定：`action_N` 的 `preSnapshot = snapshot_{N-1}`，`postSnapshot = snapshot_{N}`。

### 6.3 翻译子系统的技术方案

翻译子系统是录制数据的"消费者"，只通过文件系统读取录制子系统的产物（只读），保持完全解耦。

#### 6.3.1 预处理器（Preprocessor）—— 构建证据包

预处理器的核心价值在于：**不让大语言模型自行对比两份大段文本快照**，而是先用程序化方法将证据提取、计算、压缩为结构化的证据包（enrichedAction），再用于大语言模型推理。

预处理器依次执行以下五个子模块：

**（一）语义归并模块（Action Merge）**

对原始操作序列进行语义级别的归并，提升每条操作的语义完整性：

- **输入识别**：当操作 `action[i]` 是对输入框的点击（`click` on `input`/`textarea`），且 `action[i]` 与 `action[i+1]` 的 `formStateDelta` 中该输入框的值发生了变化时，将该操作从 `click` 重新标注为 `input`，并附加 `inputValue` 字段记录用户输入的精确文本值。此方法避免了逐字符监听键盘事件的复杂性和不可靠性。
- **密码脱敏**：对 `type="password"` 的输入框识别出的输入值自动替换为掩码标记 `[MASKED]`。
- **双击去重**：浏览器双击事件序列为 `click → click → dblclick`，本模块通过时间阈值和元素匹配检测冗余 click，标记为 `skip`，仅保留 `dblclick`。
- **噪声检测**：在后续的差异计算和表单增量计算完成后，对每条操作检查"快照差异为空 且 表单状态无变化"的条件，满足则标记为噪声操作（noise），在 AI 调用阶段跳过。

**（二）快照差异计算模块（Snapshot Diff）**

对每对相邻快照（`snapshot_{N-1}` 与 `snapshot_{N}`）执行行级差异比较，输出仅包含变化行（`+` 新增行、`-` 删除行）的精简差异文本。对超长差异文本执行截断（例如保留首尾各一部分，中间以省略标记连接），以控制提示词输入长度与计算资源消耗。

**（三）上下文片段提取模块（Context Excerpt）**

从完整的操作前快照中，根据被操作元素的特征（text、label、name、id），在快照文本行中进行模糊匹配定位，然后向上回溯找到父节点，向下收集最近的同级兄弟节点，构成操作元素附近的 UI 上下文片段。该片段保留了层级缩进关系，大幅减少 AI 需要处理的文本量，同时帮助 AI 理解元素所在的 UI 区域和语境。

**（四）表单增量计算模块（FormState Diff）**

比较相邻操作的 `formStateDelta`（当前操作与前一操作的表单状态快照），计算出值变化（changed）、新增（added）、消失（removed）三类差异，格式化为人类可读的文本摘要。该模块为 AI 提供了精确的"用户在两次操作之间修改了哪些表单元素"的证据。

**（五）操作分类与提示生成模块（Action Classify）**

根据操作类型、目标元素的 HTML 标签和属性、以及快照差异内容，程序化地进行三维度分类：

- **元素类型维度**：button、link、input、select、checkbox、switch、tab、menuitem、other
- **业务场景维度**：navigation、form-input、form-submit、dialog、toggle、selection、destructive、context-menu、other
- **操作类型维度**：click、dblclick、rightclick、keypress、input

基于分类结果，自动生成针对性的 AI 分析提示（hints），引导 AI 关注该类操作最应关注的证据。例如，对于 `form-input` 类操作，提示 AI 以 `inputValue` 为准描述输入内容；对于 `toggle` 类操作，提示 AI 关注 `checked/unchecked` 状态变化。

**预处理输出的核心数据结构 —— enrichedAction（证据包）：**

```json
{
  "index": 1,
  "type": "input",
  "originalType": "click",
  "inputValue": "admin",
  "element": { "tag": "input", "id": "username", "name": "username", ... },
  "snapshotDiff": "+ textbox \"用户名\" [value=\"admin\"]\n- textbox \"用户名\"",
  "contextExcerpt": "- form \"登录表单\"\n  - textbox \"用户名\"  ← [操作目标]\n  - textbox \"密码\"",
  "formStateChangeText": "[变化] #username: \"\" → \"admin\"",
  "classification": {
    "category": "form-input",
    "elementType": "input",
    "hints": ["这是一次文本输入操作，用户在此元素中输入了 \"admin\""]
  }
}
```

#### 6.3.2 两阶段 AI 工作流

**Phase 1：逐条操作步骤分析**

对每个 enrichedAction，构建由 System Prompt 和 User Prompt 组成的消息，调用大语言模型 API。

System Prompt 规定了严格的分析规则：
- **数据优先级**：Diff > FormStateDelta 变化 > 上下文片段 > 完整快照
- **基于证据**：每一句描述必须有操作记录或快照中的明确依据
- **禁止猜测**：不允许根据 CSS class 名或 selector 路径反推业务含义
- **描述 UI 变化而非预期结果**：只描述从 diff 中实际观察到的变化

User Prompt 按重要性分层组织所有证据（AI 分析提示 → Snapshot Diff → 表单状态变化 → 上下文片段 → 输入识别信息 → 操作基础信息 → 完整快照 → 滑动窗口上下文）。

**滑动窗口上下文机制**：每次 AI 调用携带最近 N 条（可配置）已生成的步骤描述，帮助 AI 理解操作的连续性和上下文关联。

**增量保存与中断恢复**：每条步骤分析生成后立即追加写入 `AI_steps.md` 文件。如果流程中断（网络故障、进程退出等），重新启动时会解析已有的 `AI_steps.md`，自动跳过已完成的条目，从断点处继续，避免重复的 AI 调用开销。

**Phase 2：归纳测试用例**

将 Phase 1 产出的完整步骤分析文本作为输入，调用大语言模型将操作步骤归纳为若干测试用例（Case）。归纳规则包括：同一弹窗内的操作归为一个 Case、为完成同一业务目标的连续操作归为一个 Case、页面导航后的新场景开始新 Case。最终输出结构化的测试用例表格（包含用例名称、步骤编号、操作描述、UI 变化）。

### 6.4 系统可调参数（质量/成本/速度平衡）

| 参数 | 作用 | 调节效果 |
|---|---|---|
| `SNAPSHOT_POLL_INTERVAL_MS` | 快照轮询间隔 | 越小→快照越新鲜（代价：CPU/内存开销） |
| `SNAPSHOT_MAX_DEPTH` | 快照裁剪深度 | 越深→信息越完整（代价：提示词输入长度增大） |
| `DIFF_TRUNCATE_THRESHOLD` | 差异文本截断阈值 | 越大→AI 看到越多变化细节（代价：提示词输入长度与计算资源消耗增大） |
| `EVIDENCE_CONTEXT_WINDOW_SIZE` | Phase 1 滑动窗口大小 | 越大→AI 理解更多上下文（代价：提示词输入长度与计算资源消耗增大） |

## 七、有益效果

### 7.1 解决了快照采集的竞态问题

通过将快照的"拍摄"与"保存"解耦（周期轮询 + 混合策略保存），本发明系统性地解决了现有技术中在事件处理器内异步调用快照接口导致的操作前快照污染和操作后快照不完整问题。操作前快照来自定时器轮询缓存，保证干净；操作后快照利用"下一操作到达"作为渐进行为结算的代理信号，保证完整。

### 7.2 实现了双通道互补的证据采集

异步轮询的无障碍树快照提供全页面结构化状态，同步捕获的 formStateDelta 提供表单元素的精确瞬时值。两条通道独立采集、独立使用，互相补齐对方的盲区（轮询的时间粒度限制 vs. formStateDelta 只覆盖表单元素）。

### 7.3 录制与翻译完全解耦

录制阶段只产出原始数据（操作 + 快照 + 表单状态），翻译阶段通过文件系统消费这些数据。同一份录制数据可反复翻译、调参优化，无需重新录制。这在现有的录制→脚本生成工具中是不具备的。

### 7.4 证据驱动的 AI 推理替代了 AI 自由猜测

本发明通过预处理器将原始数据转化为结构化证据包（差异、表单增量、上下文片段、分类提示），再通过 Prompt 约束强制 AI 只基于证据输出结论。与直接将大段快照文本交给 AI 自行对比相比，显著提高了输出的准确性和可验证性。

### 7.5 支持中断恢复的增量生成

Phase 1 逐条分析采用增量写盘策略，每条步骤生成后立即持久化。意外中断后可自动从断点恢复，避免重复的 AI 调用开销。对于长录制（数十到上百个操作），该特性具有显著的实用价值。

### 7.6 语义归并消除了逐字符监听的复杂性

通过比较相邻操作的 formStateDelta 来识别输入行为（而非监听每个 `keypress`/`input` 事件），本发明用一种简洁、可靠的方式解决了"用户在输入框中输入了什么"的识别问题，同时自动实现了密码脱敏。

### 7.7 零冗余的 N+1 快照存储

N 个操作仅需 N+1 张快照，相邻操作共享中间快照（`post(N) === pre(N+1)`），既节省存储空间，又保证了完整的操作前后对比能力。

## 八、具体实施方式

### 8.1 实施例 1：完整的录制-翻译流程

**步骤一：启动录制**

用户指定目标 Web 应用的 URL（如 `http://example.com/login`），系统启动 Chromium 浏览器实例，创建浏览器上下文，在上下文级别注入事件捕获脚本和回调函数桥接。拍摄初始态快照 `snapshot_000` 并保存，启动周期快照轮询（间隔 300ms）。

**步骤二：用户执行操作**

用户在浏览器中正常操作网页：
1. 点击"用户名"输入框 → 注入脚本捕获 click 事件，同时在 pointerdown capture 阶段同步采集 formStateDelta
2. 输入文本 "admin" → 无事件被捕获（不监听逐字符输入）
3. 点击"密码"输入框 → 注入脚本捕获 click 事件 + formStateDelta（此时 formStateDelta 中用户名字段已有值 "admin"）
4. 输入密码 → 无事件被捕获
5. 点击"登录"按钮 → 注入脚本捕获 click 事件 + formStateDelta

在此过程中，后台轮询持续更新缓存快照。action_2 到达时保存缓存快照为 `snapshot_001`（= post(action_1) = pre(action_2)），action_3 到达时保存缓存快照为 `snapshot_002`，以此类推。

**步骤三：停止录制**

用户关闭浏览器窗口，系统检测到 `disconnected` 事件，停止轮询，将最后的缓存快照保存为终态快照，保存 `meta.json`。

产物：3 个操作文件（`action_001.json` ~ `action_003.json`）、4 个快照文件（`snapshot_000.txt` ~ `snapshot_003.txt`）、`meta.json`。

**步骤四：预处理**

1. **语义归并**：检测到 action_1（click on input#username）到 action_2（click on input#password）之间，formStateDelta 中 `#username` 的值从 `""` 变为 `"admin"` → 将 action_1 重新标注为 `input` 类型，`inputValue = "admin"`。类似地，action_2 被标注为 `input` 类型，`inputValue = "[MASKED]"`（密码脱敏）。
2. **快照差异计算**：计算 diff_001（snapshot_000 → snapshot_001）、diff_002、diff_003。
3. **上下文提取**：为每个操作从 preSnapshot 中提取元素附近的 UI 结构片段。
4. **表单增量计算**：计算每对相邻操作之间的 formState 变化。
5. **分类标注**：action_1 → `form-input/input`，action_2 → `form-input/input`，action_3 → `form-submit/button`。

**步骤五：AI 翻译 Phase 1**

逐条调用 AI，为每个操作生成步骤分析：
- 操作 1："在用户名输入框中输入 'admin'"，依据：语义归并识别 inputValue="admin"，diff 显示 textbox 值变化。
- 操作 2："在密码输入框中输入密码（已脱敏）"，依据：语义归并识别 inputValue="[MASKED]"。
- 操作 3："点击'登录'按钮"，依据：element.tag=button, element.text="登录"，diff 显示页面跳转到首页。

增量写入 `AI_steps.md`。

**步骤六：AI 翻译 Phase 2**

将 `AI_steps.md` 全文作为输入，AI 归纳为：

```
## Case 1: 用户登录
| 步骤 | 操作 | UI 变化 |
|------|------|---------|
| 1 | 在用户名输入框中输入 "admin" | 输入框显示 "admin" |
| 2 | 在密码输入框中输入密码 | 输入框显示掩码 |
| 3 | 点击"登录"按钮 | 页面跳转到首页 |
```

输出 `AI_cases.md`。

### 8.2 实施例 2：中断恢复场景

假设用户录制了 50 个操作。Phase 1 执行到第 30 个操作时网络中断：

1. `AI_steps.md` 中已保存了前 30 条步骤分析
2. 用户重新启动翻译流程
3. 系统解析 `AI_steps.md`，识别出已有 30 条完成的分析
4. 自动从第 31 个操作开始继续，无需重复前 30 次 AI 调用
5. 全部完成后继续执行 Phase 2

**权利要求 7.** 根据权利要求 1 所述的方法，其特征在于，所述提示词输入按照预设优先级进行组织，所述优先级至少包括：页面结构快照差异优先于表单状态增量，表单状态增量优先于上下文片段；并通过系统提示词约束大语言模型仅基于所述证据包输出结论且在信息不足时输出信息不足标识。

**权利要求 8.** 根据权利要求 1 所述的方法，其特征在于，所述逐条生成自然语言操作步骤分析采用滑动窗口上下文机制，在当前操作对应的提示词输入中进一步包含最近 N 条已生成的步骤描述，以提升对操作连续性的理解。

**权利要求 9.** 根据权利要求 1 所述的方法，其特征在于，所述逐条生成自然语言操作步骤分析采用增量持久化与中断恢复策略：每条步骤分析生成后立即追加写入持久化文件；重新启动时解析所述持久化文件，识别已完成条目并从未完成条目继续生成。

**权利要求 10.** 一种基于证据驱动的 Web 用户操作录制与智能测试用例生成系统，其特征在于，包括：

录制子系统：配置有浏览器注入脚本模块、周期快照轮询引擎和分文件存储引擎，用于执行权利要求 1 所述方法中的操作记录生成、页面结构快照周期获取与缓存、表单状态快照同步采集以及混合策略保存页面结构快照；

翻译子系统：配置有预处理器和 AI 工作流引擎，所述预处理器包括语义归并模块、快照差异计算模块、上下文片段提取模块、表单增量计算模块和操作分类模块，用于执行权利要求 1 所述方法中的预处理生成证据包、调用大语言模型逐条生成自然语言操作步骤分析以及归纳生成结构化测试用例集；

所述录制子系统和翻译子系统通过文件系统解耦，录制子系统的输出目录结构作为翻译子系统的只读输入。

## 十一、摘要

本发明公开了一种基于证据驱动的 Web 用户操作录制与智能测试用例生成方法及系统。在录制阶段，通过浏览器注入脚本捕获用户物理操作，同时采用周期轮询无障碍树快照与混合策略保存相结合的方式，解决了事件处理器内异步快照调用导致的操作前快照污染和操作后快照不完整的竞态问题，实现了 N 个操作对应 N+1 个快照的零冗余存储；并在事件 capture 阶段同步采集表单状态，形成与异步快照互补的双通道证据采集机制。在翻译阶段，通过预处理器将原始录制数据转化为包含快照差异、表单增量、上下文片段、分类提示的结构化证据包，再通过两阶段 AI 工作流（逐条步骤分析 + 归纳测试用例）生成自然语言测试用例，其中 AI 被约束为仅基于证据输出结论。两阶段通过文件系统完全解耦，支持同一份录制数据反复翻译优化。
