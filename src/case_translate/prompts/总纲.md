# LLM Prompt 总纲

> 说明：本系统**只有「翻译」链路会调用 LLM**。录制、预处理、Midscene YAML、Selenium 导出均不调用。  
> 所有 Prompt 正文在 `prompts/md/`。**JSON 输出契约已合并进各 Phase 的 System Prompt（Output Format 章节）**，不再通过 API `response_format` 传递。  
> `prompts/schema/` 保留为机器可读参考副本，与 System md 同步维护。  
> LLM 全量审计落盘：`run_*/llm_audit/`（`index.json` + `problems.json` + 每条 `call_XXXX.json`）。

---

## 一、整体调用顺序

```
录制完成 (meta + actions + snapshots)
        ↓
    预处理（无 LLM）
        ↓
┌─ Phase 1 ─ 结构化步骤 ─────────────────────────────┐
│  md: phase1-structured-system / user (LangGPT 9维)  │
│  Output Format 含 parsedSteps JSON 契约              │
│  （失败时可选）md: phase1-batch-repair-*             │
│  → step_2_structured_steps.json                    │
└────────────────────────────────────────────────────┘
        ↓
┌─ Phase 2 ─ Case 归纳 ──────────────────────────────┐
│  md: phase2-case-window-system / user              │
│  → AI_cases.md                                     │
└────────────────────────────────────────────────────┘
        ↓
    Phase 3 Midscene YAML（无 LLM）
        ↓
┌─ Phase 4 ─ Agent TXT ──────────────────────────────┐
│  md: phase4-agent-txt-system / user                │
│  （失败时可选）md: phase4-agent-txt-repair-*         │
│  → case_4_agents.txt                               │
└────────────────────────────────────────────────────┘
```

**触发入口：** Dashboard「AI 翻译」、`npm run translate`、独立翻译 EXE。

## 二、Phase 1 — 结构化步骤（主 Prompt）

### 什么时候用

- 预处理完成后，对**每一条有效 action**（非 skip、非 noise）调用 LLM。
- 默认 **每批 3 条** action 合并为一次请求（微批处理）。
- 每批发送 **1 次** system + user，期望返回 **N 条**结构化步骤（N = 本批 action 数）。

### 使用的文件

| 文件 | 角色 | 何时加载 |
|------|------|----------|
| `md/phase1-structured-system.md` | System（LangGPT 9 维，含 Output Format / JSON 契约） | **每次 Phase 1 主调用必用** |
| `md/phase1-structured-user.md` | User | **每次 Phase 1 主调用必用** |

### User Prompt 占位符（代码填入，不在 md 里改）

| 占位符 | 含义 |
|--------|------|
| `{{contextHistory}}` | 本批之前最近 3 条已解析步骤的摘要 |
| `{{actionCount}}` | 本批 action 数量 |
| `{{actionBlocks}}` | 本批每条 action 的证据 JSON（element、snapshotDiff、formStateDelta 等） |

### LLM 要做什么

把物理操作 + 证据翻译成结构化字段：`description`、`uiChange`、`page`、`actionKind`、`target`、`basis`、`confidence` 等，写入 `parsedSteps[]`。

### 产物

`run_*/step_2_structured_steps.json`

### 调用参数

`temperature=0`，**无** `response_format`；JSON 契约由 System Prompt 的 **Output Format** 章节约束（MiniMax 等模型不支持 API 侧 json_schema）。

## 三、Phase 1 — 批次 JSON 修复（备用 Prompt）

### 什么时候用

- **仅当** Phase 1 主调用返回的内容 **整批无法解析**（不是 JSON、或缺少 `parsedSteps` 数组）时触发。
- 触发条件：`parsedSteps.length === 0` 且错误类型为 `batch-parse-error` 或 `batch-structure-error`。
- **不会**在「能解析但个别字段缺失」时使用（那是校验/自愈层的事，不是本 Prompt）。

### 使用的文件

| 文件 | 角色 |
|------|------|
| `md/phase1-batch-repair-system.md` | System |
| `md/phase1-batch-repair-user.md` | User（`{{rawReply}}` = 上一次失败的原始输出，截断 8000 字符） |

### LLM 要做什么

把非法输出**修成** `{ parsedSteps: [...] }` 合法 JSON，不再重新分析业务。

### 调用参数

`temperature=0`，**不带** `response_format`（纯文本修复）。

---

## 四、Phase 2 — 测试用例归纳（主 Prompt）

### 什么时候用

- Phase 1 完成后，对 **有效步骤**（`status=normal` 或 `fallback`）做 Case 归纳。
- **滑动窗口**调用：每轮取最多 20 条步骤，LLM 输出 **1 个 Case**，然后按 `consumeStepCount` 向前滑动，直到全部消费完。
- 一轮录制通常触发 **多轮** LLM 调用（例如 4 轮 → 4 个 Case）。

### 使用的文件

| 文件 | 角色 | 何时加载 |
|------|------|----------|
| `md/phase2-case-window-system.md` | System | **每轮 Phase 2 必用** |
| `md/phase2-case-window-user.md` | User | **每轮 Phase 2 必用** |

### User Prompt 占位符

| 占位符 | 含义 |
|--------|------|
| `{{windowStepsJson}}` | 当前窗口内瘦身步骤的 JSON |
| `{{indexListText}}` | 本窗口允许的 index 列表，如 `[1,2,3,...,20]` |

### LLM 要做什么

从窗口步骤中归纳 **1 个**测试用例（title、summary、步骤表），且只覆盖窗口 **前缀连续** 的一部分步骤。

### 产物

`run_*/AI_cases.md`

### 调用参数

`temperature=0.3`，无 json_schema；本地 `parseSingleCaseJsonResponse` 解析 JSON。

## 五、Phase 4 — Agent 用例聚合（主 Prompt）

### 什么时候用

- Phase 1 完成后（与 Phase 2 并行依赖 `step_2`，不依赖 Phase 2 结果）。
- 对有效步骤做 **业务逻辑聚合**，生成供下游自动化 Agent 执行的文本用例。
- **滑动窗口**：默认每窗最多 20 条；通常 **1 次**调用即可覆盖整段录制（步骤少时）。

### 使用的文件

| 文件 | 角色 | 何时加载 |
|------|------|----------|
| `md/phase4-agent-txt-system.md` | System（LangGPT 9 维，含 Output Format / JSON 契约） | **每次 Phase 4 主调用必用** |
| `md/phase4-agent-txt-user.md` | User | **每次 Phase 4 主调用必用** |

### User Prompt 占位符

| 占位符 | 含义 |
|--------|------|
| `{{stepsJson}}` | 本窗步骤精简 JSON（description、action、target、inputText、uiChange） |

### LLM 要做什么

输出 JSON：`useCaseName`、`useCasePurpose`、`agentSteps[]`（含 logicalName、microActions、consumeStepCount）。  
本地代码再把 JSON **渲染**成 `case_4_agents.txt` 纯文本。

### 产物

`run_*/case_4_agents.txt`

### 调用参数

`temperature=0.1`，无 json_schema。

---

## 六、Phase 4 — JSON 修复（备用 Prompt）

### 什么时候用

- **仅当** Phase 4 主调用返回的内容 **无法解析为 JSON** 时触发。
- 先尝试 `cleanMarkdownFence` + 暴力提取 `{...}`；仍失败才走修复 LLM。

### 使用的文件

| 文件 | 角色 |
|------|------|
| `md/phase4-agent-txt-repair-system.md` | System |
| `md/phase4-agent-txt-repair-user.md` | User（`{{rawReply}}` 截断 6000 字符） |

### LLM 要做什么

把非法输出修成 `{ useCaseName, useCasePurpose, agentSteps }` 合法 JSON。

|----|------|

## 八、Schema 文件说明（参考副本）

| 文件 | 对应 System Prompt | 是否传给 API |
|------|-------------------|--------------|
| `phase1-structured-batch.schema.json` | `phase1-structured-system.md` → Output Format | **否**（契约已写入 System） |
| `phase4-agent-txt-output.schema.json` | `phase4-agent-txt-system.md` → Output Format | **否** |

修改 JSON 契约时：**以 System md 的 Output Format 为准**，并同步更新 schema 副本供工具/校验使用。

## 九、快速对照：改哪份 Prompt 解决什么问题

| 你想优化… | 改哪个 md |
|-----------|-----------|
| 单步语义描述不准（输入/点击/UI 变化） | `phase1-structured-system.md` / `user.md` |
| LLM 合并步骤、丢 index | `phase1-structured-system.md`（N 进 N 出规则） |
| basis / 证据引用质量 | `phase1-structured-system.md` Output Format 中 basis 说明 |
| 整批返回不是合法 JSON | 先查模型兼容性；必要时改 `phase1-batch-repair-*.md` |
| Case 标题、摘要、步骤表格质量 | `phase2-case-window-system.md` |
| Case 切分太早/太晚（consume 步数） | `phase2-case-window-system.md`（规则 7、consumeStepCount） |
| Agent 用例逻辑步骤划分 | `phase4-agent-txt-system.md` |
| Agent 微观动作缺少定位信息 | `phase4-agent-txt-system.md`（DOM 定位红线） |
| Phase 4 输出带思考过程、非 JSON | `phase4-agent-txt-system.md`；修不动则 `phase4-agent-txt-repair-*.md` |
